### 1. 分级权限知识库系统（安全合规）

**Situation（情境）**:
企业知识库包含公开、内部、机密三类文档，需要根据用户权限级别控制访问范围，防止敏感信息泄露。现有系统缺乏细粒度的权限控制，所有用户都能访问全部知识库内容，存在严重的数据安全隐患。

**Task（任务）**:
- 设计并实现基于角色的访问控制（RBAC）系统
- 在向量数据库层面实现权限过滤
- 确保从文档上传到检索全链路的权限一致性
- 实现用户友好的权限管理界面

**Action（行动）**:
1. **架构设计**:
   - 扩展 Milvus Schema，新增 `kb_level`（权限级别）、`kb_category`（分类标签）、`department`（部门限制）三个标量字段
   - 设计三级权限体系：Level 1（公开）、Level 2（内部）、Level 3（机密）
   - 采用 Milvus 原生过滤表达式 `expr="kb_level <= user_level"` 实现高效权限检查

2. **核心实现**:
   - 修改 `milvus_manager.py` 的 `insert()` 方法，支持权限元数据标记
   - 在 `search()` 方法中集成 `user_level` 参数，自动添加权限过滤条件
   - 重构 5 种 RAG 检索策略（Adaptive/Hybrid/Context/Simple/Enhanced），全部支持权限传递
   - 修改 `kb_loader.py`，在文档加载阶段自动提取并标记权限信息

3. **用户体验优化**:
   - 开发登录系统（`ui/login.py`），支持用户认证和权限信息存储
   - 在侧边栏顶部显示用户权限卡片（紫色渐变背景 + 权限徽章）
   - 上传界面新增权限级别选择器，带实时权限验证（低权限用户无法上传高级别文档）
   - 实现自动分类映射（公司政策/技术文档/财务数据等 8 类）

4. **测试验证**:
   - 编写自动化测试脚本 `test_permission_system.py`，覆盖三级权限的插入、检索、过滤场景
   - 测试用例包括：Level 1 仅返回公开文档、Level 2 返回公开+内部、Level 3 返回全部

**Result（结果）**:
- ✅ **安全性提升**: 100% 的权限检查准确率，零误检零漏检
- ✅ **性能优化**: Milvus 原生过滤比后处理快 3-5 倍，检索延迟 < 100ms
- ✅ **用户体验**: 权限上传错误提示，防止误操作；权限状态实时可视化
- ✅ **可扩展性**: 支持未来扩展为 4 级、5 级权限体系，支持部门维度过滤
- ✅ **代码质量**: 完整的单元测试、详细的技术文档（`PERMISSION_SYSTEM_IMPLEMENTATION.md`）

### 2. 混合检索 RAG 系统（检索准确率）

**Situation（情境）**:
传统单一向量检索存在语义偏移问题，纯关键词匹配无法理解用户意图，导致检索召回率低（< 70%），用户体验差。企业知识库包含技术文档、政策文件、FAQ 等多种类型，需要更智能的检索策略。

**Task（任务）**:
- 构建支持语义 + 关键词的混合检索系统
- 提升检索召回率到 90% 以上
- 实现多级 RAG 优化策略（相关性检查、查询转换、重排序等）
- 支持自适应路由，根据查询类型选择最优策略

**Action（行动）**:
1. **技术选型**:
   - 向量数据库从 ChromaDB 迁移到 Milvus（支持混合检索、更高并发）
   - 嵌入模型升级为 BGE-M3（1024维 Dense + 动态维度 Sparse 双向量）
   - 采用 RRF (Reciprocal Rank Fusion) 算法融合密集和稀疏检索结果

2. **5 阶段增强 RAG 流程**:
   - **Stage 1 - 相关性检查**: 使用 LLM 判断查询是否与知识库相关（过滤无关问题）
   - **Stage 2 - 查询转换**: 生成 3 个语义变体（Multi-Query），提升召回覆盖面
   - **Stage 3 - 混合检索**: Dense（语义相似度）+ Sparse（关键词匹配）双路召回
   - **Stage 4 - 结果合并**: RRF 融合，权衡语义和精确匹配
   - **Stage 5 - 重排序**: BGE-Reranker 对 Top-20 候选进行精排，返回 Top-5

3. **自适应检索策略**:
   - 设计 `AdaptiveRAG` 模块，根据查询长度、复杂度自动选择策略
   - **Simple 策略**: 短查询（< 10 字）直接混合检索
   - **Hybrid 策略**: 中等查询（10-30 字）启用查询转换
   - **Enhanced 策略**: 复杂查询（> 30 字）启用完整 5 阶段流程
   - **Context Enriched**: 返回相邻 chunk，解决上下文碎片化问题

4. **索引优化**:
   - Dense 向量使用 IVF_FLAT 索引（nlist=128, nprobe=10），平衡速度和准确率
   - Sparse 向量使用 SPARSE_INVERTED_INDEX（倒排索引），毫秒级关键词检索
   - 配置 COSINE 相似度（归一化向量）和 IP 内积（稀疏向量）

**Result（结果）**:
- ✅ **召回率提升**: 从 68% → 92%（相比单一向量检索）
- ✅ **准确率提升**: Top-5 准确率从 75% → 88%（引入 Reranker）
- ✅ **查询延迟**: < 200ms（包含 LLM 调用和重排序）
- ✅ **用户满意度**: 内部测试反馈"答案更准确、更全面"，满意度从 3.2/5 → 4.5/5
- ✅ **技术沉淀**: 输出《TOP5_RAG 策略集成说明》技术文档，成为团队最佳实践

### 3. 多模态 Agent 系统（智能对话）

**Situation（情境）**:
传统 BI 系统需要用户掌握 SQL，且无法回答业务知识类问题。用户需要在多个系统间切换，体验割裂。需要构建一个统一的智能助手，支持自然语言查询数据库 + 知识库检索 + 数据可视化。

**Task（任务）**:
- 设计 LangGraph 状态机，支持多工具协同
- 实现自然语言到 SQL 的转换（Text-to-SQL）
- 集成数据查询、知识检索、图表生成三大能力
- 支持多 LLM 提供者切换（OpenAI/Qwen/Deepseek/Claude）

**Action（行动）**:
1. **Agent 架构设计**:
   - 使用 LangGraph 构建状态机，定义 4 种系统提示模式：
     - 完整模式（DB + KB）、数据分析模式（DB only）、对话模式（KB only）、纯对话模式
   - 实现动态系统提示选择，根据 `session_state` 中的资源状态自动切换
   - 设计工具调用链：`text_to_sql → execute_sql → generate_chart` 强制执行

2. **Text-to-SQL 实现**:
   - 开发 `tools_text2sql.py`，使用 LLM 将自然语言转换为 SQL
   - 引入数据字典（`data_dictionary.txt`），提供表结构和业务规则上下文
   - Prompt 工程：Few-shot 示例 + 错误处理 + SQL 验证规则
   - 支持复杂查询（JOIN、聚合、子查询、窗口函数）

3. **数据可视化**:
   - 开发 `tools_charts.py`，自动生成 Highcharts 配置 JSON
   - 支持 8 种图表类型：柱状图、折线图、饼图、散点图、面积图、组合图、热力图、漏斗图
   - Agent 强制调用图表工具：数据查询后必须输出 ```json 代码块
   - 前端使用 Streamlit + Highcharts.js 渲染交互式图表

4. **多 LLM 支持**:
   - 实现工厂模式（`llm_factory.py`），支持 7 种 LLM 提供者
   - OpenAI、Qwen、Deepseek、Claude、Ollama、ZhipuAI、Baichuan
   - 统一接口封装，支持运行时切换（无需重启应用）
   - 外观模式（`llm_manager.py`）提供全局访问点

5. **会话管理**:
   - 使用 Streamlit `session_state` 管理对话历史
   - 实现 Agent 重新初始化逻辑：DB/KB 状态变化时自动重建
   - 支持对话历史持久化（可扩展到数据库存储）

**Result（结果）**:
- ✅ **用户效率提升**: 数据查询时间从 5 分钟 → 30 秒（无需手写 SQL）
- ✅ **功能整合**: 一个界面完成数据分析 + 知识问答 + 可视化，减少 70% 系统切换
- ✅ **LLM 灵活性**: 支持 7 种 LLM，成本可控（Deepseek 比 GPT-4 便宜 95%）
- ✅ **可视化质量**: 自动生成专业图表，无需手动配置 Highcharts 参数
- ✅ **错误处理**: SQL 执行失败自动重试，工具调用失败降级为纯对话

**Action（行动）**:

4. **自动化测试**:
